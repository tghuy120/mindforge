# 控制论与科学方法论——从控制论到 AI Agent 设计方法论

> 本文基于金观涛、华国凡《控制论与科学方法论》阅读笔记，结合与 ChatGPT 讨论，探索控制论核心概念在 AI Agent 设计与 Harness Engineering 中的迁移应用。
>
> 相关笔记：「西方本体论的当代转折」（哲学方法论层面的互补视角，待创建）

## 一、核心命题：Agent = 负反馈控制系统 + 共轭变换 + 随机探索

这是本次阅读的核心收获——控制论的三个基本机制，恰好对应 AI Agent 的三个设计支柱：

| 控制论概念 | Agent 对应 | 作用 |
|-----------|-----------|------|
| **负反馈** | 评估-修正循环 | 让系统逐步逼近目标（收敛） |
| **共轭变换** | 问题表示与拆解 | 把不可操作的问题变成可操作的形式 |
| **随机探索** | 多路径尝试与策略切换 | 跳出局部最优，避免死循环 |

这不是类比，而是结构同构——AI Agent 本质上就是一个控制系统。

---

## 二、控制论的逻辑主线

书中的核心逻辑可以压缩为一条主线：

```
现实世界（不可控）
    ↓ L（共轭变换）
可操作表示空间
    ↓ Agent（负反馈 + 探索）
控制策略
    ↓ L⁻¹（逆变换）
作用回世界
    ↓
误差（feedback）→ 循环
```

这条主线揭示了一个关键洞察：**大多数人做 Agent / Harness 工程，失败的原因不是模型不行，而是没有统一的控制框架，导致系统是"拼出来的"，不是"设计出来的"。**

## 三、五个关键概念的工程映射

### 3.1 可能性空间（Problem Space）

控制论的第一步不是行动，而是**定义可能性空间**——状态空间、动作空间、目标空间。这是 99% 的人忽略的。

以 Coding Agent 为例：
- **State**：当前代码状态
- **Action**：修改代码 / 调用工具
- **Goal**：测试通过

> **方法论原则**：不建模可能性空间，不要写 Agent。

### 3.2 共轭变换（Conjugate Transformation）

共轭变换是控制论中最深刻的概念之一：**把一个在原始空间中难以解决的问题，变换到另一个空间中去解决，再变换回来**。

在 Harness 工程中，这就是表示层（Representation Layer）的设计：

```
现实世界 → L → 模型输入（prompt / structured data）
模型输出 → L⁻¹ → 现实动作（tool call / code change）
```

常见的变换类型：

| 变换类型 | 用途 |
|---------|------|
| text | 推理与规划 |
| JSON | 工具调用 |
| embedding | 语义搜索 |
| AST | 代码操作 |

> **关键判断**：如果 Agent 很"聪明"但做不对事，90% 是共轭变换设计错了——不是推理问题，是表示问题。
>
> **方法论原则**：先设计 L（共轭变换），再写 prompt。"不要问用什么 prompt，要问世界是怎么被编码的。"

### 3.3 负反馈（Negative Feedback）

负反馈是 Agent 的"灵魂"。核心公式：

```
error = goal - current_state
```

没有可计算的 error，就没有系统。反馈有三个等级：

| 类型 | 示例 | 强度 |
|------|------|------|
| binary | pass / fail | 弱 |
| scalar | score 分值 | 中 |
| structured | 多维指标 | 强 |

越往后越稳定。

关于正反面例子的平衡：在 Agent 设计中，不能给太多反面例子（"不要做 X"），否则 Agent 变得患得患失。**需要正面示范引导收敛方向**，这与控制论中"目标函数应该是正向定义的"一致。

> **方法论原则**：没有 error function，就没有系统。不要追求一次成功，要允许逐步逼近。

### 3.4 随机探索（Exploration）

随机探索在 Harness 中的意义不是让 Agent 随机行动，而是**受控的偏离**——在确定性策略失效时，有计划地切换路径。

三种探索机制：

| 类型 | 例子 | 场景 |
|------|------|------|
| retry | 重试当前方案 | 偶发失败 |
| branch | 多方案并行 | 不确定最优路径 |
| strategy shift | 换方法 | 当前方向根本不通 |

探索不是 Model 层面的 temperature 采样，而是 Harness 层面的**逃逸机制**（escape mechanism）：什么时候放弃当前路径？什么时候换策略？这必须显式设计。

> **方法论原则**：探索必须有触发条件。至少要有一个 escape 机制。

### 3.5 控制能力（Control Capacity）

控制能力 = 系统能把可能性空间缩小多少。

```
控制能力 = 信息 × 工具 × 反馈 × 表示
```

Agent 失败 ≠ 模型不行，90% 是控制能力不够——信息不足、工具不足、反馈太弱、表示不当。

> **方法论原则**：所有复杂问题都是控制能力不足的问题。

## 五、十条方法论硬规则

从控制论到 Agent 设计，提炼出的可迁移原则：

1. **不建模可能性空间，不要写 Agent**
2. **先设计 L（共轭变换），再写 prompt**
3. **没有 error function，就没有系统**
4. **不要追求一次成功，要允许逐步逼近**
5. **探索必须有触发条件**
6. **工具不是能力，使用方式才是**
7. **表示错误 > 推理错误**（大多数失败是表示问题）
8. **所有复杂问题都是控制能力不足**
9. **避免不可逆操作**（控制论中的"陷阱"概念）
10. **系统必须能自我修正**

## 六、思想地图

```
                    [现实世界]
                        ↓
                [可能性空间定义]  ← 决定问题难度
                        ↓
                  [共轭变换 L]   ← 决定问题是否可解
                        ↓
                   [表示空间]
                        ↓
               [Agent 控制系统]
               ├── 负反馈        ← 决定是否收敛
               ├── 探索          ← 决定是否能跳出局部最优
               └── 策略选择      ← 决定执行路径
                        ↓
              [动作执行 L⁻¹]
                        ↓
                  [现实反馈]
                        ↓
                     （循环）
```

概念关系：
- 可能性空间 → 决定问题难度
- 共轭变换 → 决定是否可解
- 负反馈 → 决定是否收敛
- 探索 → 决定是否能跳出局部最优
- 控制能力 → 决定成功概率

## 七、三套框架的对比——控制论五层 vs 六层架构 vs 三层控制模型

先给出六层架构和三层控制模型的参照图，再逐一对比分析。

![统一 Harness 视图——六层架构](../asset/harness-六层架构.png)

![三层控制模型——系统的动力学](../asset/harness-三层控制模型.png)

### 7.1 三套框架各自回答什么问题

| 框架         | 视角         | 核心问题             | 设计导向  |
| ---------- | ---------- | ---------------- | ----- |
| **控制论五层**  | 设计原理（WHY）  | 为什么 Agent 要这样设计？ | 理论完备性 |
| **六层架构**   | 工程结构（WHAT） | 系统需要哪些模块？        | 模块完备性 |
| **三层控制模型** | 运行动力学（HOW） | 模块间如何闭环运转？       | 系统收敛性 |

一句话总结：**六层搭骨架，三层调行为，控制论给理论根基。搭系统用六层，调系统用三层，设计决策回控制论。**

### 7.2 映射关系

#### 控制论五层 → 六层架构

| 控制论层 | 映射到六层 | 说明 |
|---------|----------|------|
| Layer 1 可能性空间 | L1 Policy + L2 Planning | Policy 定边界（缩小可能性空间），Planning 定义 State/Action/Goal |
| Layer 2 共轭变换 | L3 Skills + L4 Execution | Skills 是变换能力的注册（L 的工程实现），Execution 中的工具调用是 L⁻¹ 逆变换 |
| Layer 3 负反馈 | L6 Eval | Eval 层就是负反馈的工程载体——error 计算、质量门禁、人类接管触发 |
| Layer 4 探索 | **分散，无显式层** | 回滚在 L4，重规划在 L2，触发在 L6——六层架构没有独立的"探索层" |
| Layer 5 控制能力 | 六层整体的"元属性" | 不是某一层，而是所有层协同的综合能力 |
| **（缺位）** | **L5 Runtime** | 状态追踪、上下文管理、资源隔离、检查点快照——控制论五层没有显式建模"状态基础设施"，但它是负反馈获取 current_state、探索回滚到检查点的前提 |

#### 控制论五层 → 三层控制模型

| 控制论层 | 映射到三层 | 说明 |
|---------|----------|------|
| Layer 1 可能性空间 | Constraint（约束系统） | 约束 = 缩小可能性空间的工程手段 |
| Layer 2 共轭变换 | 横跨 Constraint + Execution | Constraint 定义编码规则（L），Execution 执行变换（L⁻¹） |
| Layer 3 负反馈 | Cognition → Constraint 闭环 | 认知系统评估结果，反馈回约束系统调整策略 |
| Layer 4 探索 | Cognition 的"失败分析/策略调整" | 探索由认知系统触发——识别到"当前路径不通"后切换策略 |
| Layer 5 控制能力 | 收敛性公式 | = 约束强度 × 反馈频率 / 执行自由度 |

### 7.3 关键洞察

**1. 控制论五层是"正交维度"，不是"串行层级"**

六层架构和三层模型都是模块/层级划分——模块之间有边界、有接口。但控制论五层更像是**每个模块都需要考虑的设计维度**。比如"共轭变换"不只存在于 L3 Skills，而是 L2 的任务分解（把复杂问题变换为可拆解的子任务）、L3 的能力抽象（把工具变换为可组合的能力）、L4 的工具调用（把模型输出变换为现实动作）中都有体现。

**2. 控制论五层解释了三层动力学"为什么能收敛"**

三层模型给出了收敛性公式：`收敛性 = 约束强度 × 反馈频率 / 执行自由度`。但没有解释每个因子从何而来。控制论五层给出了答案：

- **约束强度** ← 可能性空间定义的精确度 + 共轭变换的质量
- **反馈频率** ← 负反馈系统的设计（binary/scalar/structured，反馈间隔）
- **执行自由度** ← 探索机制的受控程度（完全自由 vs 受控偏离）

**3. 六层架构中"探索"是隐式的——这是一个改进点**

控制论明确将探索作为独立机制（与负反馈并列的第二支柱）。但六层架构中，探索分散在 L4（回滚）、L2（重规划）、L6（反馈触发重试）三层中，没有显式的设计位置。这意味着：
- 很容易在搭建六层时**忘记设计探索机制**
- retry/branch/strategy-shift 三种探索没有统一的管理点
- **建议**：在六层架构中，将探索机制作为 L4 Execution 的显式子系统，或在 L2 Planning 中增加"重规划触发条件"的显式设计

**4. 认知系统是三套框架共同指向的最薄弱环节**

- 控制论：负反馈 + 自我修正能力是系统收敛的前提
- 三层模型：Cognition（L6 Eval + L3 Skills）"当前最薄弱"
- 六层架构：L6 Eval 是金字塔顶端，最后建设，往往最不完善

三者一致指向：**Agent 系统失败的主因不是执行能力不足，而是认知/评估/反馈闭环缺失。**

**5. 控制论五层缺少"状态观测器"——L5 Runtime 的映射缺位**

六层工程模型中的 L5 Runtime（状态追踪、上下文管理、资源隔离、检查点快照）在控制论五层中找不到显式对应。这不是偶然遗漏，而是控制论的经典理论假设"状态是已知的"（`error = goal - current_state` 中，current_state 被当作给定量），但在工程中**谁来维护 current_state** 恰恰是最重的基础设施：

- **负反馈**需要 `current_state` → 由 Runtime 的状态追踪提供
- **共轭变换**需要维护表示空间 → 由 Runtime 的上下文管理提供
- **探索**需要回滚能力 → 由 Runtime 的检查点快照提供
- **可能性空间**需要边界隔离 → 由 Runtime 的资源隔离提供

L5 Runtime 是控制系统的**状态观测器（State Observer）**——不是某一层的映射目标，而是所有控制机制的运行基础。**建议**：在控制论框架中增设「L0 状态观测层」，或将 Runtime 作为贯穿五层的正交维度。

### 7.4 综合视图

```
         控制论（WHY）          六层架构（WHAT）         三层模型（HOW）
         ═══════════          ═══════════════         ═══════════════
         可能性空间    ───→    L1 Policy          ─┐
                              L2 Planning         ─┤→  Constraint
                                                   │   （约束系统）
         共轭变换      ───→    L3 Skills          ─┤→  ─────────────
                              L4 Execution        ─┤→  Execution
                                                   │   （执行系统）
         （⚠ 缺位）          L5 Runtime          ─┤→  ─────────────
         负反馈        ───→    L6 Eval             ─┤→  Cognition
         探索          ───→    （分散在 L2/L4/L6）  ─┘   （认知系统）

         控制能力      ===     六层整体完备性       ===   收敛性公式

         L5 Runtime = 状态观测器（State Observer），是所有控制机制的运行基础
```

## 八、反思与下一步

### 身份转变

这本书（以及这次讨论）指向的方向是：**从"AI 使用者"到"控制系统设计者"**。写 prompt 是使用，设计 Harness 是工程，而控制论提供的是工程背后的理论框架。

### 两个风险

1. **停留在概念层**——每个概念必须对应代码结构，否则只是"知道"而非"掌握"
2. **过度简化映射**——"共轭 = step 拆分"、"探索 = sampling" 这种降维理解会丢失控制论的精髓

### 下一步

- 拿一个正在做的 Agent（如 yoga-guru-copilot-platform 中的某个模块），用 Harness Design Sheet 完整拆解一次
- 对照五层框架，找出当前设计中缺失的层（反馈？共轭？探索？）
- 将控制论视角融入 [[meta-harness]] wiki 概念页
